# NoviqWiki Content Format

> [English](CONTENT_FORMAT.md) | [简体中文](CONTENT_FORMAT.md#简体中文)

Markdown is the canonical editable format for NoviqWiki pages. Published history is immutable; editor drafts are mutable working records. NoviqWiki v0.1.0 does not implement MediaWiki syntax, templates, extensions, migration behavior, or MediaWiki API compatibility.

## Drafts, Pages, and Revisions

A page record holds the current title, slug, lifecycle status, protection level, and current published revision pointer.

Saving a draft does not create a revision. It upserts one `page_drafts` row for the page/editor pair with:

- Page ID.
- Editor ID.
- Base revision ID.
- Markdown source.
- Edit summary.
- Created and updated timestamps.

Publishing a page or rolling back creates a new immutable `page_revisions` row. A stored revision contains:

- Revision ID and page ID.
- Parent revision ID and monotonically increasing revision number.
- Canonical Markdown source.
- Sanitized rendered HTML.
- Searchable/readable plain text.
- Content hash.
- Nullable editor ID plus preserved editor display name.
- Edit summary and publication state.
- Derived headings, category names, and outbound wiki-link target titles.
- Created timestamp.

Rename, archive, protection changes, and soft deletion change page metadata without rewriting historical revisions. To change published content, create another revision; never update an old revision in place.

## Supported Markdown

The renderer currently supports:

- CommonMark paragraphs, headings, thematic breaks, lists, blockquotes, links, and images.
- GitHub Flavored Markdown tables, strikethrough, autolinks, and task lists.
- Inline code and fenced code blocks.
- Syntax highlighting for languages recognized by `rehype-highlight`.
- Inline and block math through `remark-math` and KaTeX.
- Heading IDs generated by `rehype-slug`.
- NoviqWiki wiki links and category declarations described below.

Raw HTML in Markdown is disabled. MediaWiki markup such as `== Heading ==`, templates such as `{{Template}}`, parser functions, tags/extensions, and MediaWiki categories are not interpreted.

## Standard Links and Wiki Links

Use ordinary Markdown for explicit URLs:

```markdown
[Getting started](/page/getting-started)
[External site](https://example.com)
```

Use wiki links when the target is a page title:

```markdown
[[Getting Started]]
[[Getting Started|start here]]
[[入门指南]]
[[入门指南|从这里开始]]
```

The text before `|` is normalized as a page title and converted to `/page/{slug}`. The text after `|` is the visible label. Published relationship metadata keeps one outbound relationship per normalized target.

At article-render time, an existing published target receives its current slug. A missing/draft/archived/deleted target receives the missing-link style; a user with `page.create` can follow it to a pre-filled new-page form.

The current wiki-link preprocessor uses a regular expression over the entire Markdown string before Markdown parsing. It therefore also recognizes `[[...]]` inside fenced/inline code and may rewrite it. Avoid literal double-bracket examples in page code samples, or escape/break the brackets, until wiki-link parsing becomes AST-aware.

## Categories

Declare a category with the English, case-insensitive `Category:` prefix:

```markdown
[[Category:Operations]]
[[Category:运维]]
```

`[[分类:运维]]` is not recognized by the current parser. The category name after `Category:` may be Chinese or English, but the namespace keyword must remain `Category`.

Category declarations are removed from rendered article HTML. When a revision is published, its categories replace the page's current `page_categories` projection in the same transaction as links and search metadata. Historical revisions retain their own category-name arrays.

## Redirect Pages

Make the first non-empty source line a complete redirect directive:

```markdown
#REDIRECT [[Target Page]]
```

or:

```markdown
#重定向 [[目标页面]]
```

The keyword is case-insensitive for `REDIRECT`. A UTF-8 BOM and blank lines before the directive are tolerated. The parser accepts an optional target section or label but resolves the page-level target.

Redirect pages retain their own immutable history. Normal article browsing follows content redirects server-side and reports the redirect origin. Edit, history, backlinks, and maintenance routes can address the source without following it. Publication and rollback reject detectable loops; runtime resolution also detects loops and defaults to a maximum depth of eight hops.

`/redirects` lists published redirect pages and distinguishes valid, double, missing, draft, archived, and deleted targets.

## Content Maintenance Pages

Current published link/category projections drive these reports:

- `/wanted`: missing targets referenced by published pages.
- `/orphaned`: published pages with no inbound link from another published page.
- `/dead-end`: published pages with no link to another published page.
- `/short-pages`: published non-redirect pages at or below a selected plain-text length.
- `/protected-pages`: published pages with edit protection.
- `/uncategorized`: published pages without category membership.
- `/redirects`: published content redirects and target status.
- `/special`: grouped entry points for browsing, maintenance, and authorized administration.

Draft, archived, and deleted source pages are excluded from these public maintenance projections unless a report explicitly describes their target status.

## Media and Attachments

Upload media through the UI or `POST /api/v1/media`. With local storage, use the returned `publicUrl`. With S3, an upload-time signed `publicUrl` is time-limited; durable page content should use the same-origin `/media/{storageKey}` route built from the returned storage key. Storage URLs contain generated site/year/random keys, so do not assume the original filename is a stable `/media/{filename}` route.

Embed an image with descriptive alternative text:

```markdown
![Architecture diagram](/media/site-id/2026/generated-key.png)
```

Link non-image attachments such as PDF files:

```markdown
[Download the operations handbook](/media/site-id/2026/generated-key.pdf)
```

The paths above illustrate the returned local URL shape only. Copy the actual `publicUrl` from the media response or media picker.

The renderer allows sanitized HTTP(S) image sources, but the application Content Security Policy uses `img-src 'self' data: blob:`. Use same-origin uploaded media for reliable display; an arbitrary remote image can be removed by browser CSP even when its markup survives sanitization.

Uploads enforce a site size limit, a MIME allowlist, SVG rejection, filename normalization, randomized storage keys, and path containment. When `file-type` cannot identify the bytes, current validation falls back to the client-declared MIME type. Keep a narrow allowlist and do not treat an extension or declared MIME type as proof of content.

## Code Blocks

Use a fenced block and a language identifier when highlighting is desired:

````markdown
```ts
export function title(value: string) {
  return value.trim();
}
```
````

Recognized languages receive highlight classes. Unknown languages remain readable plain code. As noted above, literal wiki-link syntax is preprocessed even inside code in v0.1.0.

## Math

Use `$...$` for inline math and `$$...$$` for block math:

```markdown
Inline: $a^2 + b^2 = c^2$

$$
E = mc^2
$$
```

KaTeX output is generated before sanitization and must pass the same safe HTML schema.

## Rendering and Sanitization

The publication renderer performs this pipeline:

```text
wiki-link/category extraction
 -> remark-parse
 -> remark-gfm
 -> remark-math
 -> remark-rehype (allowDangerousHtml: false)
 -> rehype-slug
 -> rehype-katex
 -> rehype-highlight
 -> rehype-sanitize
 -> rehype-stringify
```

The sanitizer removes disallowed tags, attributes, event handlers, and URL protocols. Stored article HTML is rendered by the article component only after this pipeline. Do not pass untrusted or unsanitized HTML to React `dangerouslySetInnerHTML`.

## Search Text and Current Projections

Plain text is derived from sanitized rendered HTML by removing tags, decoding common entities, collapsing whitespace, and trimming. Published pages copy the current title, aliases, plain text, and category names into `search_index`.

Search uses PostgreSQL's `simple` text-search configuration, weighted fields, prefix queries, and case-insensitive substring fallbacks. The search index and current link/category tables are rebuildable projections; they never replace canonical Markdown or immutable revision history.

## Titles, Slugs, and Heading IDs

Page titles are trimmed, must contain `1..220` characters, and may not contain `<`, `>`, `[`, `]`, `{`, `}`, `|`, or `#`.

Slugs are derived from a supplied slug or title by:

1. Applying Unicode NFKD normalization and lowercase conversion.
2. Removing combining accents.
3. Keeping ASCII letters/digits and CJK Unified Ideographs in `\u4e00-\u9fff`.
4. Replacing other runs with `-`, trimming/deduplicating hyphens, and falling back to `page` if empty.

This means many scripts outside ASCII and the supported CJK range can collapse to a poor or duplicate slug. Supply and review an explicit slug for those titles. A title, page slug, or alias that conflicts inside the site is rejected. Renaming can keep the old slug as an alias.

Heading IDs are generated from rendered heading text and deduplicated within a document. Changing heading text can change its anchor. Prefer the stable page route for durable links and use heading fragments only when that coupling is acceptable.

## Publishing Workflow

1. Create or open the page in the Markdown editor.
2. Save a per-user draft while work is incomplete.
3. Preview the rendered result; preview does not create a revision.
4. Include a concise edit summary.
5. Publish against the current `baseRevisionId`. A stale base returns a conflict instead of overwriting another publication.
6. Review history/diff. Rollback creates another revision rather than mutating history.

Publishing updates category membership, wiki links/backlinks, and search metadata transactionally. Draft saves do not update those public projections.

## Unsupported or Important Edge Cases

- Raw HTML, MediaWiki markup/templates/extensions, Mermaid, and a MediaWiki-compatible API are not supported.
- Chinese redirect keyword `#重定向` is supported; Chinese category namespace `分类:` is not.
- Wiki-link/category scanning is not Markdown-AST-aware and can affect code samples.
- Remote images can be blocked by the application CSP.
- Embedded video/iframe markup is not part of the supported Markdown contract.
- Drafts are mutable and per editor; only published revisions are immutable history.
- Content redirects and rename aliases are different: redirects are authored in Markdown, while aliases are current slug mappings.

## Authoring Checklist

- Use one top-level heading per page when practical.
- Use descriptive link text instead of a raw URL.
- Use wiki links for title-based internal relationships and Markdown links for explicit routes/files.
- Add useful alt text to every meaningful image.
- Copy media URLs from the product; do not invent storage paths.
- Keep edit summaries short and specific.
- Use tables only for genuinely tabular comparisons.
- Preview math, tables, code, Chinese text, and long links before publishing.
- Do not paste arbitrary HTML or secrets into page source.

## 简体中文

> [English](CONTENT_FORMAT.md) | [简体中文](CONTENT_FORMAT.md#简体中文)

Markdown 是 NoviqWiki 页面可编辑内容的规范格式。已发布历史不可变，编辑者草稿是可变工作记录。NoviqWiki v0.1.0 不实现 MediaWiki 语法、模板、扩展、迁移行为或 MediaWiki API 兼容性。

### 草稿、页面与修订

页面记录保存当前标题、路径名、生命周期状态、保护级别和当前已发布修订指针。

保存草稿不会创建修订。它会为“页面/编辑者”组合更新或插入一条 `page_drafts` 记录，包含：

- 页面 ID。
- 编辑者 ID。
- 基础修订 ID。
- Markdown 源文。
- 编辑摘要。
- 创建和更新时间戳。

发布页面或回滚会创建新的不可变 `page_revisions` 记录。已存储修订包含：

- 修订 ID 和页面 ID。
- 父修订 ID 和单调递增的修订编号。
- 规范 Markdown 源文。
- 净化后的渲染 HTML。
- 可搜索/可读纯文本。
- 内容哈希。
- 可空编辑者 ID，以及保留的编辑者显示名称。
- 编辑摘要和发布状态。
- 派生标题、分类名称和出站 Wiki 链接目标标题。
- 创建时间戳。

重命名、归档、保护变更和软删除只改变页面元数据，不重写历史修订。要更改已发布内容，应创建新修订；绝不要原地更新旧修订。

### 支持的 Markdown

当前渲染器支持：

- CommonMark 段落、标题、分隔线、列表、引用、链接和图片。
- GitHub Flavored Markdown 表格、删除线、自动链接和任务列表。
- 行内代码和围栏代码块。
- `rehype-highlight` 能识别语言的语法高亮。
- 通过 `remark-math` 和 KaTeX 支持行内及块级数学公式。
- 由 `rehype-slug` 生成标题 ID。
- 下文所述的 NoviqWiki Wiki 链接和分类声明。

Markdown 中的原始 HTML 已禁用。`== 标题 ==` 等 MediaWiki 标记、`{{模板}}`、解析器函数、标签/扩展和 MediaWiki 分类不会被解释。

### 标准链接与 Wiki 链接

对明确地址使用普通 Markdown：

```markdown
[入门](/page/getting-started)
[外部网站](https://example.com)
```

目标是页面标题时使用 Wiki 链接：

```markdown
[[Getting Started]]
[[Getting Started|start here]]
[[入门指南]]
[[入门指南|从这里开始]]
```

`|` 前的文本会按页面标题规范化并转换为 `/page/{slug}`；`|` 后的文本是可见标签。已发布关系元数据会为每个规范化目标保留一条出站关系。

渲染文章时，已存在且已发布的目标使用当前路径名。缺失/草稿/已归档/已删除目标使用缺失链接样式；拥有 `page.create` 的用户可以打开预填的新页面表单。

当前 Wiki 链接预处理器会在 Markdown 解析前，对完整 Markdown 字符串使用正则表达式。因此，它也会识别围栏/行内代码中的 `[[...]]`，并可能改写它。在 Wiki 链接解析改为感知 AST 前，应避免在页面代码示例中直接写双中括号，或转义/拆开括号。

### 分类

使用英文且不区分大小写的 `Category:` 前缀声明分类：

```markdown
[[Category:Operations]]
[[Category:运维]]
```

当前解析器不识别 `[[分类:运维]]`。`Category:` 后的分类名称可以是中文或英文，但命名空间关键词必须保留为 `Category`。

分类声明会从渲染文章 HTML 中移除。发布修订时，其分类会在与链接和搜索元数据相同的事务中替换页面当前 `page_categories` 投影。历史修订保留各自的分类名称数组。

### 重定向页面

让第一个非空源文行成为完整重定向指令：

```markdown
#REDIRECT [[Target Page]]
```

或：

```markdown
#重定向 [[目标页面]]
```

`REDIRECT` 关键词不区分大小写。指令前允许 UTF-8 BOM 和空行。解析器接受可选的目标小节或标签，但解析到页面级目标。

重定向页面保留自身不可变历史。普通文章浏览会在服务器端跟随内容重定向，并显示重定向来源。编辑、历史、反向链接和维护路由可以不跟随跳转而直接访问来源。发布和回滚会拒绝可检测循环；运行时解析同样检测循环，默认最大深度为八跳。

`/redirects` 列出已发布重定向页面，并区分有效、双重、缺失、草稿、已归档和已删除目标。

### 内容维护页面

当前已发布链接/分类投影驱动以下报告：

- `/wanted`：已发布页面引用的缺失目标。
- `/orphaned`：没有其他已发布页面入站链接的已发布页面。
- `/dead-end`：没有链接到其他已发布页面的已发布页面。
- `/short-pages`：不超过所选纯文本长度的已发布非重定向页面。
- `/protected-pages`：启用编辑保护的已发布页面。
- `/uncategorized`：没有分类成员关系的已发布页面。
- `/redirects`：已发布内容重定向及目标状态。
- `/special`：公开浏览、维护和已授权管理入口的分组索引。

除非报告明确描述目标状态，否则这些公开维护投影会排除草稿、已归档和已删除来源页面。

### 媒体与附件

通过 UI 或 `POST /api/v1/media` 上传媒体。本地存储应使用返回的 `publicUrl`。使用 S3 时，上传时生成的签名 `publicUrl` 有时限；长期页面内容应根据返回的存储键使用同源 `/media/{storageKey}` 路由。存储地址包含生成的站点/年份/随机键，因此不要假设原文件名对应稳定的 `/media/{filename}` 路由。

使用有意义的替代文本嵌入图片：

```markdown
![架构图](/media/site-id/2026/generated-key.png)
```

链接 PDF 等非图片附件：

```markdown
[下载运维手册](/media/site-id/2026/generated-key.pdf)
```

以上路径只用于说明返回的本地地址形状。应从媒体响应或媒体选择器复制真实 `publicUrl`。

渲染器允许净化后的 HTTP(S) 图片来源，但应用内容安全策略使用 `img-src 'self' data: blob:`。为确保可靠显示，应使用同源上传媒体；即使任意远程图片标记通过净化，浏览器 CSP 仍可能阻止它。

上传会执行站点大小限制、MIME 允许列表、SVG 拒绝、文件名规范化、随机存储键和路径限制。当 `file-type` 无法识别字节时，当前校验会回退到客户端声明的 MIME 类型。应维持严格允许列表，不要把扩展名或声明 MIME 类型视为内容证明。

### 代码块

需要高亮时，使用围栏代码块和语言标识符：

````markdown
```ts
export function title(value: string) {
  return value.trim();
}
```
````

可识别语言会获得高亮 class。未知语言仍显示为可读纯代码。如上所述，v0.1.0 即使在代码中也会预处理字面 Wiki 链接语法。

### 数学公式

行内数学使用 `$...$`，块级数学使用 `$$...$$`：

```markdown
行内：$a^2 + b^2 = c^2$

$$
E = mc^2
$$
```

KaTeX 输出在净化前生成，并必须通过相同的安全 HTML Schema。

### 渲染与净化

发布渲染器执行以下流水线：

```text
Wiki 链接/分类提取
 -> remark-parse
 -> remark-gfm
 -> remark-math
 -> remark-rehype（allowDangerousHtml: false）
 -> rehype-slug
 -> rehype-katex
 -> rehype-highlight
 -> rehype-sanitize
 -> rehype-stringify
```

净化器会移除不允许的标签、属性、事件处理器和 URL 协议。只有经过此流水线后，文章组件才渲染已存储 HTML。不要把不可信或未经净化的 HTML 传给 React `dangerouslySetInnerHTML`。

### 搜索文本与当前投影

纯文本从净化后的渲染 HTML 派生：移除标签、解码常见实体、合并空白并去除首尾空白。已发布页面把当前标题、别名、纯文本和分类名称复制到 `search_index`。

搜索使用 PostgreSQL 的 `simple` 文本搜索配置、字段加权、前缀查询和不区分大小写的子字符串回退。搜索索引及当前链接/分类表都是可重建投影；它们绝不替代规范 Markdown 或不可变修订历史。

### 标题、路径名与标题 ID

页面标题会去除首尾空白，长度必须为 `1..220` 个字符，且不得包含 `<`、`>`、`[`、`]`、`{`、`}`、`|` 或 `#`。

路径名从传入路径名或标题按以下规则派生：

1. 应用 Unicode NFKD 规范化并转换为小写。
2. 移除组合重音符。
3. 保留 ASCII 字母/数字和 `\u4e00-\u9fff` 内的 CJK 统一表意文字。
4. 把其他连续字符替换成 `-`，去除/合并连字符；结果为空时回退为 `page`。

这意味着 ASCII 和受支持 CJK 范围之外的许多文字可能生成较差或重复的路径名。此类标题应提供并检查明确路径名。站点内冲突的标题、页面路径名或别名会被拒绝。重命名时可以把旧路径名保留为别名。

标题 ID 根据渲染标题文本生成，并在文档内去重。更改标题文本可能改变锚点。需要长期稳定时优先使用页面路由；只有能接受这种耦合时才使用标题片段。

### 发布工作流

1. 在 Markdown 编辑器中创建或打开页面。
2. 内容未完成时保存每用户草稿。
3. 预览渲染结果；预览不会创建修订。
4. 填写简短编辑摘要。
5. 基于当前 `baseRevisionId` 发布。基础修订过期会返回冲突，不会覆盖他人发布。
6. 检查历史/差异。回滚会创建新修订，而不是修改历史。

发布会在事务中更新分类成员关系、Wiki 链接/反向链接和搜索元数据。保存草稿不会更新这些公开投影。

### 不支持或需要注意的边界情况

- 不支持原始 HTML、MediaWiki 标记/模板/扩展、Mermaid 和 MediaWiki 兼容 API。
- 支持中文重定向关键词 `#重定向`；不支持中文分类命名空间 `分类:`。
- Wiki 链接/分类扫描不感知 Markdown AST，可能影响代码示例。
- 远程图片可能被应用 CSP 阻止。
- 嵌入视频/iframe 标记不属于受支持 Markdown 契约。
- 草稿可变且按编辑者隔离；只有已发布修订属于不可变历史。
- 内容重定向与重命名别名不同：重定向写在 Markdown 中，别名是当前路径名映射。

### 编写检查表

- 适用时每页只使用一个一级标题。
- 使用有意义的链接文本，不直接显示原始地址。
- 基于标题的内部关系使用 Wiki 链接，明确路由/文件使用 Markdown 链接。
- 为每张有意义图片添加有效替代文本。
- 从产品复制媒体地址，不要自行编造存储路径。
- 编辑摘要保持简短、具体。
- 只有真正的表格比较才使用表格。
- 发布前预览数学公式、表格、代码、中文文本和长链接。
- 不要把任意 HTML 或机密粘贴到页面源文。
